## What "wired into the chat graph" means

CLAUDE.md's Orchestrator Routing table says every intent should route to its agent through the chat graph's conditional edges (`ai_service/graph/orchestrator.py`). As of 2026-07-09, `_INTENT_TO_NODE` has real edges for `eligibility_query` (Agent 1), `comparison` (Agent 8), `financial_plan` (Agent 7), `document_verify` (Agent 4, guidance-only), `application_request` (Agent 3), `grievance` (Agent 5), `csc_assist` (Agent 9), `status_check` (a real node — see below), and `small_talk` (a dedicated greeting node — no scheme retrieval). The placeholder node is now reached **only** by the injection guard's `blocked` pseudo-intent (reply already set) or an unknown/future intent. This same graph now backs **all three transports** — REST, text WebSocket, and real-time voice — so voice gets the identical 12-agent intelligence as text, not a separate weaker flow.

**`status_check` wired 2026-07-09** (`ai_service/graph/agents/status_check.py`). It was the last intent still hitting the placeholder ("Status check abhi Spring Boot gateway se hoga"). Now it reads the citizen's OWN application rows directly from the shared `applications` collection (Spring Boot's model; `userId == citizen_id == JWT sub`), read-only — creating/mutating an application stays a Spring REST concern — and returns a grounded per-application status summary in Hinglish (submitted / saved / approved / rejected / disbursed → friendly phrases, externalAppId surfaced when present). Grounded-first: the reply is built deterministically from the real rows (works with zero LLM budget, can't hallucinate a status), with an optional LLM warmth pass that falls back to the deterministic text on any error. Verified end-to-end against real Mongo (seeded 3 real applications → correct newest-first summary; zero-applications and empty-citizen edges handled) + 9 pure-function unit tests in `ai_service/tests/test_status_check.py` (suite now 71). One check deferred to a services-up session: a live chat turn exercising the LLM intent classifier's `status_check` classification (the routing itself is deterministically confirmed).
